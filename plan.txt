

# Web App 개발 Plan

## 1. 전체 구조

```
사용자 브라우저→ 로그인→ 재고 검색 화면→ 서버 API→ Google Sheets 재고 데이터
```

초기 버전은 **Google Sheets를 데이터 원본**으로 사용합니다.

## 2. 기술 구성

```
Frontend: HTML + CSS + JavaScriptBackend: Node.js 또는 Python FastAPIData: Google SheetsLogin: 자체 로그인Hosting: Render / Railway / Google Cloud Run / Vercel
```

## 3. 데이터 컬럼

Google Sheet 컬럼은 아래처럼 고정합니다.

```
품명규격신품/중고무자료가격계산서가격입고일자제조사 판매가재고수량
```

추가 권장 컬럼:

```
ID수정일시비고
```

## 4. 로그인 기능

사용자 테이블을 별도 Sheet 또는 서버 DB에 둡니다.

```
아이디비밀번호 해시값권한사용여부마지막 로그인
```

비밀번호는 평문 저장 금지입니다.  
반드시 해시 처리합니다.

권한 예시:

```
admin: 전체 관리viewer: 재고 조회만 가능
```

## 5. 재고 검색 기능

검색창에 입력하면 실시간으로 필터링합니다.

검색 방식:

```
"모터" 입력 → 품명/규격/제조사에서 모터 포함 항목 표시"삼성 AND 신품" → 두 조건 모두 포함"삼성 OR LG" → 둘 중 하나 포함"모터 AND 중고 OR 펌프" → 조건식 검색
```

초기 버전은 클라이언트에서 전체 재고 데이터를 받은 뒤 브라우저에서 검색하는 방식이 빠릅니다.

```
로그인 성공→ 재고 데이터 전체 다운로드→ 브라우저 메모리에 저장→ 검색창 입력 시 즉시 필터링
```

## 6. 화면 구성

```
[로그인 화면]아이디비밀번호로그인 버튼[재고 조회 화면]검색창AND / OR 검색 안내새로고침 버튼재고 테이블로그아웃 버튼
```

테이블 컬럼:

```
품명 | 규격 | 신품/중고 | 무자료가격 | 계산서가격 | 입고일자 | 제조사 판매가 | 재고수량
```

## 7. 보안 구조

브라우저에서 Google Sheets에 직접 접근하지 않습니다.

```
브라우저→ 내 서버 API→ Google Sheets API
```

중요 정보는 서버에만 둡니다.

```
Google 서비스 계정 키Sheets ID사용자 비밀번호 해시
```

## 8. API 설계

```
POST /api/loginGET /api/inventoryPOST /api/logoutGET /api/me
```

관리자 수정 기능이 필요하면 나중에 추가합니다.

```
POST /api/inventory/updatePOST /api/inventory/sync
```

## 9. 캐시 방식

재고 수정은 하루 1회 정도이므로 서버에서 캐시합니다.

```
서버 시작 또는 첫 요청→ Google Sheets에서 재고 로드→ 서버 메모리에 캐시→ 사용자는 캐시 데이터 조회→ 관리자 새로고침 시 Sheets 재로드
```

이렇게 하면 접속자가 많아도 Google Sheets API 호출이 적습니다.

## 10. 개발 단계

### 1단계: Google Sheet 정리

- 컬럼명 확정
- 샘플 데이터 입력
- 서비스 계정 생성
- Sheet 읽기 권한 부여

### 2단계: 백엔드 개발

- 로그인 API
- 세션/JWT 처리
- Google Sheets 읽기
- 재고 데이터 JSON 변환
- 캐시 기능

### 3단계: 프론트엔드 개발

- 로그인 화면
- 재고 테이블
- 실시간 검색
- AND / OR 검색
- 로그아웃

### 4단계: 배포

- 서버 배포
- HTTPS 적용
- 환경변수 설정
- 접속 테스트

### 5단계: 운영 기능 추가

- 관리자 계정 관리
- 검색 결과 엑셀 다운로드
- 재고수량 0 이하 표시
- 최근 입고순 정렬
- 모바일 화면 최적화

## 최종 추천 구조

```
HTML Web App+ 자체 로그인+ 서버 API+ Google Sheets+ 서버 캐시+ 브라우저 실시간 검색
```
